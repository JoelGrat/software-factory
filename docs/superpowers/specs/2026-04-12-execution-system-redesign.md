# Execution System Redesign

**Date:** 2026-04-12
**Branch:** feat/bulletproof-pipeline
**Status:** Approved — ready for implementation plan

---

## Problem

The current execution system is a brute-force retry loop. It has three critical failure modes:

1. **No failure routing** — type errors, test failures, and infrastructure failures are all treated identically: retry the full iteration.
2. **No error surfacing** — TypeScript diagnostics are logged as a count only. The UI shows a spinner with no actionable information.
3. **Broken commit logic** — the orchestrator commits unconditionally, regardless of whether checks passed. Users receive green-looking commits on broken code.
4. **No budget enforcement** — infinite retry on stuck conditions is possible.
5. **UI is opaque** — a single spinner tells the user nothing about what is happening or why.

---

## Architecture

The execution system gains three new layers on top of the existing iteration loop:

```
runPipeline() → executePhase()
  └── iterationLoop(budget)
        ├── implementChanges()
        ├── staticValidationPhase()       ← tsc + eslint + schema/codegen
        │     └── inlineRepair()          ← NEW: mechanical errors only
        │           └── rerun validation
        ├── testPhases(mode)              ← NEW: unit → integration → smoke
        │     └── repairPhase()           ← NEW: reasoning failures
        ├── stuckDetector()               ← NEW: pattern detection, not one rule
        └── commitPolicy()                ← NEW: explicit outcome states
```

---

## Section 1 — Failure Router

Every failure is routed by type before any repair is attempted. This prevents mechanical fixes from being wasted on architectural problems, and vice versa.

| Failure Type | First Response | Escalation |
|---|---|---|
| TypeScript error | Inline repair | Repair phase |
| ESLint / lint | Inline repair | Repair phase |
| Schema/codegen validation | Inline repair | Repair phase |
| Unit test failure | Repair phase | Human |
| Integration test failure | Repair phase | Human |
| Behavioral / E2E | Repair phase | Human |
| Build config | Repair phase | Human |
| Infra / network | Retry command | Human |

**Inline repair** = a focused prompt scoped to the exact failing files/lines. Fast, mechanical, no broad reasoning required.

**Repair phase** = a coordinated multi-file reasoning pass with full failure context. Used when pattern, architecture, or assumptions may be wrong.

## Section 2 — Orchestrator

### Budget (explicit scope)

```ts
interface ExecutionBudget {
  global: {
    maxIterations: number           // default: 5
    maxRuntimeMs: number            // default: 600_000 (10 min)
  }
  perIteration: {
    maxInlineRepairs: number        // default: 3
    maxRepairPhaseAttempts: number  // default: 2
  }
}
```

Loop exits with `status: 'budget_exceeded'` when any cap is hit. Escalates to human — never loops further.

### Static validation: observe vs generate

Two distinct sub-phases:

- **Observe-only** (pure validation): `tsc`, `eslint`, import cycle checks. Must not mutate files.
- **Generate-then-validate**: codegen runs first, then validates output. Mutations from codegen are tracked separately so the stuck detector does not conflate "file touched by repair" with "file generated by codegen."

### Test phases: short-circuit by default

```ts
type TestMode = 'fail_fast' | 'collect_all'
```

- `fail_fast` (default): unit fail → repair immediately, skip integration + smoke.
- `collect_all`: run all phases, aggregate failures before repair. Used when full diagnostics are needed.

Sequence: unit → integration → smoke. Cheap signal before expensive signal.

### Stuck detector

Fires before `commitPolicy()` on each iteration. Escalates to repair phase (or human) immediately if any of:

- Same diagnostic appears in consecutive iterations
- Error count increased vs previous iteration
- Same file patched 3+ times across iterations
- Alternating diagnostic pattern detected (A→B→A)
- Per-iteration budget hit (`maxInlineRepairs` or `maxRepairPhaseAttempts`)

### Repair confidence metadata

Every `inlineRepair` and `repairPhase` call records:

```ts
interface RepairAttempt {
  phase: 'inline' | 'repair_phase'
  filesPatched: string[]
  diagnosticsTargeted: string[]
  confidenceScore: number             // 0.0 – 1.0 (stored)
  confidenceLabel: 'high' | 'medium' | 'low'  // derived for UI
  rationale: string                   // one sentence, max 140 chars
}
```

### Commit policy

```ts
type CommitOutcome =
  | { type: 'green' }                          // all checks passed
  | { type: 'wip'; reason: string }            // changed files, still failing — explicit wip message
  | { type: 'no_commit'; reason: string }      // no diff, or infra-only failure
  | { type: 'blocked' }                        // human approval required

// WIP commit message format:
// "wip: <change title> (tsc failed: N errors)"
```

**Status transition rules for `execution_runs.status`:**

```
running → success
running → wip
running → blocked
running → budget_exceeded
running → cancelled
```

No other transitions are valid. Enforced in app layer.

### Execution summary (persisted on loop exit)

```ts
interface ExecutionSummary {
  status: 'success' | 'wip' | 'budget_exceeded' | 'blocked' | 'cancelled'
  iterationsUsed: number
  repairsAttempted: number
  filesChanged: string[]
  finalFailureType: string | null
  commitOutcome: CommitOutcome
  durationMs: number
}
```

## Section 3 — Event Schema + DB

### `execution_runs`

```sql
create table execution_runs (
  id         uuid primary key default gen_random_uuid(),
  change_id  uuid not null references change_requests(id) on delete cascade,
  status     text not null default 'running'
             check (status in ('running','success','wip','budget_exceeded','blocked','cancelled')),
  summary    jsonb,
  started_at timestamptz not null default now(),
  ended_at   timestamptz
);

create index on execution_runs (change_id, started_at desc);
```

### `execution_events`

```sql
create table execution_events (
  id             uuid primary key default gen_random_uuid(),
  run_id         uuid not null references execution_runs(id) on delete cascade,
  change_id      uuid not null references change_requests(id) on delete cascade,
  seq            bigint not null,
  iteration      int not null,       -- 0 for pre-loop events
  event_type     text not null,
  phase          text,
  schema_version int not null default 1,
  payload        jsonb not null default '{}',
  created_at     timestamptz not null default now(),

  unique (run_id, seq)
);

create index on execution_events (run_id, seq);
create index on execution_events (change_id, run_id, created_at);
```

### Event taxonomy (closed vocabulary)

Constrained in app layer via `as const` union type.

```

```

### Payload registry (per event type)

```ts
// phase.*.failed
{
  diagnostics: { file: string; line: number; message: string; code: string }[],  // first 20
  totalCount: number,
  truncated: boolean,
  durationMs: number
}

// repair.inline.* / repair.phase.*
{ attempt: RepairAttempt, durationMs: number }

// iteration.stuck
{
  reason: 'repeated_diagnostic' | 'error_count_increased' | 'same_file_repeated'
        | 'alternating_diagnostic' | 'budget_hit'
}

// phase.skipped
{ phase: string; reason: string }

// commit.wip / commit.skipped / commit.failed
{ reason: string, durationMs: number }

// all completion events
{ durationMs: number }

// execution.completed
{ summary: ExecutionSummary }
```

## Section 4 — UI

### Live strip

Pinned below the `ChangeStepBar`. Five slots with dynamic labels.

```
● Implementing   |  ○ Static validation  |  ○ Unit tests  |  ○ Integration  |  ○ Smoke checks
  Patching Sidebar.tsx...                                                         01:12  [Cancel]
```

- Labels are dynamic: "Unit tests" when unit phase active, "Integration tests" when integration phase active, etc.
- Active slot shows a one-line detail subtext (current file, test count, or repair target)
- Each slot has icon + text label — not color alone
- Strip has `role="status"` and `aria-live="polite"`
- Strip disappears when run ends, replaced by final summary banner

**Cancel button states:**
```
[Cancel]                          → default
[Cancelling…]                     → request in flight
[Cancelled]                       → confirmed
[Cannot cancel — committing]      → locked during git operation
[Force stop failed]               → infrastructure error
```

**Color semantics (supplemental to icons and labels):**

| Color | Meaning |
|---|---|
| Blue | Running |
| Green | Passed |
| Yellow | Repairing |
| Red | Blocked |
| Gray | Queued / skipped |

### Iteration cards

Emitted when `iteration.completed` fires. Latest expanded on mount, all prior collapsed.

**Collapsed:** iteration number · status badge (icon + label) · duration

**Expanded:**
- Files implemented
- Static validation: first N diagnostics with file/line; truncation notice if `truncated: true`
- Repair: `Inline · High (0.82) · "Fixed prop type mismatch on SidebarItem"`
- Test results per phase: unit ✓ / integration skipped / smoke n/a
- Commit outcome: green commit hash / wip message / skipped reason
- Actions: `View files changed` · `View patch` · `Open commit ↗` (when commit exists)

### Final state banner

Replaces live strip when run exits:

```
✓ Done in 3 iterations · 2 files changed · 4 repairs · 01:23
⚠ Blocked — stuck detector fired. Same diagnostic repeated 2×. Review required.
✗ Budget exceeded — 5 iterations used. No changes committed.
– Cancelled after 2 iterations.
```

### Error panel

Full-width, in-page (never modal). Shown when `status` is `blocked`, `budget_exceeded`, or `cancelled`.

Contains:
- Final failure type + human-readable explanation
- Last diagnostics (with truncation if needed)
- Files patched in last attempt
- Suggested action ("Review iteration 3 for the repeated diagnostic")

### Polling behavior

| State | Interval |
|---|---|
| Run active | 2s |
| Run complete | Stop, snapshot final state |
| Tab hidden | 10s |
| Tab refocused | Resume 2s immediately |
| Reconnect | Re-fetch latest `run_id`, resume from last `seq` |

### Empty state

```
No executions yet.
Run this change to see live progress, iteration history, and repair evidence.
[Run execution →]
```

## Section 5 — Concurrency Control

**Policy: single active run per `change_id`.**

When a new run is requested while one is already `running`:
- **Reject** with HTTP 409 and message: "An execution is already in progress for this change."
- The UI surfaces this as a non-modal inline error — not a spinner.

Queuing runs is explicitly out of scope for this iteration.

**Implementation:** Before inserting an `execution_runs` row, the API handler checks:

```sql
select id from execution_runs
where change_id = $1 and status = 'running'
limit 1
```

If a row exists → reject. Use a DB-level advisory lock or `insert ... where not exists` to prevent races between concurrent requests.

## Section 6 — Cancellation Semantics

Cancellation is **cooperative until a command boundary**. The orchestrator checks a `cancelled` signal between phases, not mid-command.

**Rules:**
- The cancel request sets a flag in DB (`execution_runs.cancellation_requested = true`).
- The orchestrator polls this flag at each phase boundary (after each `implementChanges`, `staticValidationPhase`, `testPhases` call).
- On detection: exit loop cleanly, set `status = 'cancelled'`, emit `execution.cancelled` event.
- **No commit after cancellation** — `commitPolicy()` is skipped entirely on cancel path.
- **Hard kill only** if the process exceeds a grace period (30s) after cooperative cancel — kills the child process directly.
- If cancellation arrives while already in `commitPolicy()` (mid-git-op): let the commit finish, then set status `cancelled`. A partial commit is worse than an unexpected one.

**Button state → backend state mapping:**

| UI State | Backend State |
|---|---|
| [Cancel] clicked | `cancellation_requested = true` set |
| [Cancelling…] | Waiting for phase boundary |
| [Cancelled] | `status = 'cancelled'`, run ended |
| [Cannot cancel — committing] | `commitPolicy()` in progress |
| [Force stop failed] | Hard kill timed out |

## Section 7 — Idempotency and Recovery

**Problem:** if the server restarts mid-run, `execution_runs.status` stays `'running'` forever.

**Recovery policy (runs on server startup and on a periodic cron):**

```
SELECT id FROM execution_runs
WHERE status = 'running'
  AND started_at < now() - interval '15 minutes'
  AND (last_heartbeat_at IS NULL OR last_heartbeat_at < now() - interval '2 minutes')
```

Stale runs → set `status = 'blocked'`, write summary with `finalFailureType: 'server_interrupted'`, emit `execution.blocked` event.

**Heartbeat:** the orchestrator writes `last_heartbeat_at = now()` to `execution_runs` every 30 seconds. Add `last_heartbeat_at timestamptz` column.

**Phase idempotency:** each phase checks whether its events already exist in `execution_events` for the current `run_id` + `iteration` before executing. If found, skip (treat as already completed). This makes a crashed-then-resumed run safe without re-running work.

## Section 8 — Security Boundaries for Repair

The repair system may autonomously modify files. It must be constrained.

**Allowed directories (whitelist — configurable per project):**
- `app/`, `components/`, `lib/`, `tests/`, `styles/`
- Root config files: `tsconfig.json`, `tailwind.config.*`, `next.config.*`

**Blocked files (hard deny, regardless of config):**
- `.env`, `.env.*`, `.env.local`
- `supabase/migrations/**` — migrations are append-only; repair must never touch them
- `package.json`, `package-lock.json` — no dependency installs without explicit approval
- Any file matching `*.pem`, `*.key`, `*.secret`

**Per-repair limits:**
- Max files patched per inline repair: 3
- Max files patched per repair phase: 8
- No shell commands beyond the defined check commands (`tsc`, `eslint`, test runner)
- No `npm install` / `pnpm add` unless explicitly unlocked in project settings

**Enforcement:** validated in the repair prompt construction layer — blocked paths are stripped from context and the model is instructed not to emit patches for them. Patches are also validated against the allowlist before being written to disk.

## Section 9 — Metrics

Track per-run and aggregate. Required for knowing whether the redesign works.

**Per-run (stored in `execution_summary`):** already covered — iterations used, repairs attempted, files changed, duration, commit outcome.

**Aggregate metrics (queryable from `execution_runs` + `execution_events`):**

| Metric | Source |
|---|---|
| Run success rate | `count(status=success) / count(*)` on `execution_runs` |
| WIP rate | `count(status=wip) / count(*)` |
| Avg iterations per run | `avg(summary->>'iterationsUsed')` |
| Top failure types | `count` by `finalFailureType` |
| Inline repair success % | `repair.inline.succeeded / (repair.inline.started)` from events |
| Repair phase success % | `repair.phase.succeeded / (repair.phase.started)` |
| Stuck detector frequency | `count(iteration.stuck)` events |
| Avg phase duration | `avg(payload->>'durationMs')` grouped by `event_type` |

No separate analytics table needed for MVP — queries run against `execution_runs` and `execution_events` directly. Add materialized views if query time degrades.

## Section 10 — Event Retention

Append-only logs grow fast. Retention policy:

| Data | Retention |
|---|---|
| `execution_events` full rows | 90 days, then delete |
| `execution_runs` + `summary` | Indefinite |
| Diagnostic payloads > 50KB | Compress (store as gzip base64 in payload, `compressed: true` flag) |

Implement as a scheduled DB job (Supabase cron or `pg_cron`). Runs nightly.

## Section 11 — Schema Validation Ownership

TypeScript types are not enforcement. Runtime validation at the write boundary is.

**Policy:** every `execution_events` insert goes through a Zod schema keyed by `event_type`. If payload does not match the registered schema → log error, reject insert, emit `infra.retrying` or surface to operator.

**Implementation pattern:**

```ts
const EVENT_PAYLOAD_SCHEMAS: Record<EventType, z.ZodSchema> = {
  'phase.static_validation.failed': z.object({
    diagnostics: z.array(diagnosticSchema).max(20),
    totalCount: z.number(),
    truncated: z.boolean(),
    durationMs: z.number(),
  }),
  // ... one schema per event type
}

function insertEvent(event: RawEvent) {
  const schema = EVENT_PAYLOAD_SCHEMAS[event.event_type]
  const parsed = schema.safeParse(event.payload)
  if (!parsed.success) throw new EventPayloadValidationError(event.event_type, parsed.error)
  // ... insert
}
```

## Section 12 — Commit Safety

**Additional rule:** never commit if the working tree contains pre-existing changes unrelated to this run.

Before `commitPolicy()` executes:

```bash
git status --porcelain
```

If any modified/untracked files exist that are not in `filesChanged` for this run → abort commit, set outcome `no_commit`, surface as a warning in the UI: "Working tree contains unexpected changes. Commit skipped to prevent contamination."

This prevents the agent from accidentally committing unrelated dirty state from the host environment.

## What is NOT in scope

- E2E test infrastructure (modeled in failure router, not implemented)
- Token budget tracking
- Multi-agent parallel execution
- Replay / re-run of a specific iteration
- Run queuing (single active run enforced; queue is a future addition)

These may be added in a follow-on spec.
